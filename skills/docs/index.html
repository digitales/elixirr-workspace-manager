Golden Rule

Do not refresh memory from raw exports first.

Raw comms should be normalized into `outputs/` before they influence `working-memory/current.md`.

  1. `manual-exports/`
  2. `elixirr-comms-normalizer`
  3. `outputs/slack` or `outputs/teams`
  4. `elixirr-memory-refresh`
  5. `working-memory/current.md`

Operating Model

Five layers with distinct jobs

The system works best when each folder type does one thing well. This keeps refresh logic predictable and makes the automations easier to trust.

01

`context/`

Durable truth: client background, project framing, decisions, people, sources.

02

`working-memory/`

Live operational state: what matters now, what changed, what is blocked, what happens next.

03

`meetings/`

Structured records of conversations, decisions, action items, and transcripts.

04

`outputs/`

Processed summaries from automations, Slack, Teams, and agent-generated artifacts.

05

`manual-exports/`

Raw copied or exported source material waiting to be normalized or archived.

For Non-Technical Users

Use Codex by asking for outcomes, not by memorising skills

A non-technical teammate should not need to think about scripts, folders, or automation config. The easiest model is: tell Codex what you want done, give it the source material, and decide whether it should happen once or keep happening automatically.

Ask

Describe the outcome in plain English

  • “Set up a new client workspace for Mastercard.”
  • “Turn this transcript into proper meeting notes.”
  • “Process the latest Slack export for this client.”
  • “Refresh the working memory for this project.”
  • “Draft the follow-up message from this meeting.”
Provide

Give Codex the minimum source material it needs

  • A client name or project name
  • A transcript, rough notes, or exported Slack / Teams file
  • The audience, if you want a follow-up message or email
  • The instruction “make this automatic” if it should run in the background
Trust

Codex picks the right underlying skill

The user-facing request stays simple. Behind the scenes, Codex decides whether this is a workspace setup job, meeting-note job, comms-processing job, memory-refresh job, or publishing job.

Automate

Use automations for repeatable jobs

If the same task will happen again and again, ask Codex to keep doing it. That is when the recurring automations become useful: checking drop zones, normalising new exports, refreshing memory, or syncing outputs elsewhere. 

What do you want done?
What source material do you have?
Should this happen once, or automatically from now on?

Setup

Recommended rollout for a new team

Start small: install the structure, create one client and one project, then wire the automations that feed the memory loop.

Step 1

Download and extract the package

Start by downloading the latest package zip from GitHub Releases and extracting it anywhere locally. This gives the team the skills, docs, automations, and installers in one portable folder.

Step 2

Run the installer

Use the installer that matches your platform. The Elixirr workspace root is typically `~/Documents/elixirr` on macOS and `%USERPROFILE%\Documents\elixirr` on Windows.

macOS / Linux
cd /path/to/extracted-package
bash ./skills/scripts/install.sh

Windows PowerShell
cd C:\path\to\extracted-package
pwsh -File .\skills\scripts\install.ps1
Step 3

Document the durable context

Fill the entrypoints first so downstream meeting notes and memory refreshes inherit the right framing.

  • `clients/<client>/context/index.md`
  • `projects/<project>/context/index.md`
  • `people.md`, `preferences.md`, `sources.md`, `decisions.md`
Step 4

Wire the intake paths

Give raw source material a predictable landing zone.

  • `raw-meetings/` for unfiled transcripts
  • `manual-exports/slack/` and `manual-exports/teams/` for raw comms
  • `outputs/automations/` for processed external automation results
Step 5

Turn on the recurring jobs

These two automations demonstrate the intended cadence once the team is ready for background workflows.

  • `elixirr-raw-meeting-dropzone` every 15 minutes
  • `elixirr-output-sync` hourly with local automation-to-client mappings

Installation

How the skills are installed in practice

Codex uses runtime-installed copies from the local `.codex` home, and the live automations point at those installed paths when they run.

Package

Local skill package

This package is designed to be downloaded, extracted locally, and used in place. It contains the skill definitions, docs, and installer scripts used to refresh a local Codex install.

  • `skills/<skill-name>/` contains each skill definition
  • `skills/scripts/` contains the macOS/Linux and Windows installers
  • `automations/<automation-name>/automation.toml` contains local automation templates when present
Runtime

Installed Codex copies

The live runtime copies sit under the local Codex home.

  • macOS / Linux: `~/.codex/skills/<skill-name>/` and `~/.codex/automations/<automation-name>/automation.toml`
  • Windows: `%USERPROFILE%\.codex\skills\<skill-name>\` and `%USERPROFILE%\.codex\automations\<automation-name>\automation.toml`
  • Automation prompts reference installed scripts under the local Codex home `skills/.../scripts/` path
Refresh

Installer scripts for each platform

The simplest team path is to download the package, extract it, open a terminal in that folder, and use the installer script for their platform. Under the hood, both installers follow the same refresh model: copy updated skill folders or files into the local Codex skills directory, then copy or update the matching automation TOMLs in the local Codex automations directory. 

cp -R skills/elixirr-workspace-manager ~/.codex/skills/
cp -R skills/elixirr-raw-meeting-dropzone ~/.codex/skills/
cp automations/elixirr-raw-meeting-dropzone/automation.toml \
  ~/.codex/automations/elixirr-raw-meeting-dropzone/automation.toml
macOS / Linux
cd /path/to/extracted-package
bash ./skills/scripts/install.sh
bash ./skills/scripts/install.sh --dry-run

Windows PowerShell
cd C:\path\to\extracted-package
pwsh -File .\skills\scripts\install.ps1
pwsh -File .\skills\scripts\install.ps1 -DryRun

Signal Flow

The automation loop in plain English

The team does not need to memorize every script. They need to understand the handoff points and which skill owns each transition.

Capture

Raw inputs arrive

Meeting transcripts land in `raw-meetings/`. Slack and Teams exports land in `manual-exports/`. External automations write processed markdown into a staging directory.

Structure

Skills turn inputs into durable records

`elixirr-raw-meeting-dropzone`, `elixirr-meeting-writer`, and `elixirr-comms-normalizer` create clean meeting notes and output summaries in the expected folders.

Promote

Working memory gets refreshed

`elixirr-memory-refresh` promotes only live signals into `working-memory/current.md`, keeping the file concise and current.

Act

Humans and agents work from the same state

Follow-up comms, decisions, priorities, and next actions can now be generated from a shared, trustworthy baseline. 

raw transcript -> elixirr-raw-meeting-dropzone -> meetings/
raw Slack export -> elixirr-comms-normalizer -> outputs/slack/
automation markdown -> elixirr-automation-output-publisher -> outputs/automations/
meetings/ + outputs/ -> elixirr-memory-refresh -> working-memory/current.md

Foundation

The umbrella operational skill for scaffolding, meeting shells, channel workspaces, and automation templates.

Use it when
You need the multi-tool router for setup, helper scripts, and file placement.
Owns

Capture

elixirr-meeting-notes

Turns a transcript or rough notes into the standard Elixirr meeting note structure.

Use it when
The destination file exists or the filing path is already understood.
Produces
Summary, Decisions, Action Items, Risks / Blockers, References, Transcript

Capture

elixirr-meeting-writer

The paste-driven end-to-end workflow from transcript to saved finished note.

Use it when
A human pastes a transcript and wants the filing plus drafting handled in one pass.
Special behavior
Asks only short follow-up questions when filing risk is real.

Capture

elixirr-raw-meeting-dropzone

Batch-processes markdown transcripts from `raw-meetings/`, files them, and archives the originals.

Use it when
You want folder-driven transcript ingestion instead of pasting content into chat.
Automation fit
Best for recurring cron runs over the drop zone.

Capture

elixirr-comms-normalizer

Converts raw Slack and Teams exports into concise processed summaries under `outputs/`.

Use it when
Raw communication files are sitting in `manual-exports/` and should become reusable signals.
Protects against
Refreshing memory from noisy, unstructured raw source material.

Memory

elixirr-memory-refresh

Updates `working-memory/current.md` from the latest meetings, automation outputs, and communication summaries.

Use it when
The workspace already has processed inputs and needs the live state refreshed.
Promotes
Current Focus, Active Priorities, Risks / Blockers, Next Actions, Latest Signals

Memory

elixirr-memory-bootstrap

Builds the first useful baseline for a legacy client or project with lots of existing history.

Use it when
You are migrating into the v2 model and need a current-state summary without replaying everything forever.
Differs from refresh
Chunked historical summarization first, then a clean `working-memory/current.md` baseline.

Delivery

elixirr-follow-up-comms

Turns saved meeting notes into outbound Slack, Teams, email, or task-list artifacts.

Use it when
The team needs a post-meeting message or action list grounded in the note.
Can save to
`outputs/communications/` when the user wants durable drafts.

Delivery

elixirr-automation-output-publisher

Copies processed automation markdown from local staging areas into Elixirr `outputs/automations/` folders.

Use it when
An external automation can only write inside its own local workspace or staging area.
Enables
Multiple automation feeds to participate in the same memory refresh pipeline.

Automation Examples

Demo automations plus the wider live chain

The two highlighted examples are still the best show-and-tell anchors, but the live `.codex` install also reveals the broader production pattern around them.

Cron every 15 minutes

Elixirr Raw Meeting Dropzone

Watches the `raw-meetings` folder inside the Elixirr workspace root, infers client and meeting metadata, saves a structured meeting note, and archives the raw transcript only after success.

  • Best for recurring transcript intake
  • Typical roots: `~/Documents/elixirr/raw-meetings` on macOS or `%USERPROFILE%\Documents\elixirr\raw-meetings` on Windows
  • Routes unmatched files to `internal/meetings/`
  • Pairs `workspace-manager` helpers with `meeting-notes` formatting

Cron every hour

Elixirr Output Sync

Pulls processed markdown from local automation output folders into the right client `outputs/automations/` destination without overwriting same-name files.

  • Best for external local automations
  • Uses local `--mapping NAME SOURCE DESTINATION` definitions
  • Keeps the shared automation template generic and reusable

Client Comms Normalize

Hourly client-specific jobs run `elixirr-comms-normalizer` over unarchived Slack and Teams exports.

Client Memory Refresh

Hourly refresh jobs promote processed comms, meetings, and automation outputs into current working memory.

Team Guidance

What good usage looks like

Use setup skills first

Get the folders and context files right before adding automations or memory refresh logic.

Ask for the job, not the implementation

Most teammates should describe the outcome they want and let Codex choose the skill or automation path.

Normalize before promoting

Raw Slack or Teams material should become processed summaries before it influences working memory.

Keep working memory current, not exhaustive

`working-memory/current.md` should read like the latest operating picture, not an archive dump.

Archive only after success

Whether it is a transcript or a communication export, the raw source should remain in place until the durable output exists.
